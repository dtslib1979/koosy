## Claude Code 웹의 구조적 한계

### 로직/알고리즘 작업
```
입력 → 처리 → 출력
   ↑___ 자체 검증 가능 ___↓
```
- 코드 실행 결과 예측 가능
- 테스트로 검증 가능
- 정답이 명확함
- **효율: 높음**

### 웹 디자인 작업
```
CSS 작성 → 렌더링 → ???
              ↓
         시각 피드백 불가
              ↓
         유저 스크린샷 대기
              ↓
         추측해서 수정
              ↓
         반복 (삽질)
```
- **효율: 낮음**

---

## 디자인 작업이 어려운 구체적 이유

| 문제 | 설명 |
|------|------|
| 피드백 루프 끊김 | 렌더링 결과를 볼 수 없음 |
| 문맥 의존성 | "블록 안에" → 어떤 클래스인지 추측 필요 |
| 시각적 판단 불가 | "예쁘다/못생겼다" 판단 불가 |
| CSS 연쇄 효과 | 한 줄 변경의 다른 영향을 볼 수 없음 |
| 공간 관계 파악 불가 | z-index 겹침, 여백 등 시각적 확인 필요 |

## 효율적 지시 방법 (디자인 작업 시)

### 비효율적 지시
```
❌ "블록 안에 채워"
❌ "더 크게"
❌ "저기 수정해"
❌ "느낌이 별로야"
```

### 효율적 지시
```
✅ ".graph-container 안에 ::before로 채워"
✅ "40px → 80px로 변경"
✅ "line 1285 .logo 클래스 수정"
✅ 스크린샷 + "이 요소는 .card-wrapper임"
```

## 디자인 작업 시 삽질 최소화 체크리스트

```
□ 클래스명/셀렉터 명시
□ 수치 변경은 구체적 픽셀값 (예: 40px → 80px)
□ 스크린샷 + 해당 요소 클래스명 같이 제공
□ pseudo-element 작업 시 부모 요소 명시
□ "더 크게/작게" 대신 배수 또는 픽셀값
```

## 🎯 핵심 차별점 3가지

### 1. 생산 속도 10배
```
기존: 경험 → 메모 → PPT → 촬영 → 편집 → 업로드 (며칠~일주일)
우리: 경험 → Parksy Capture → 화면녹화 → 업로드 (수십 분)

편집? 없음.
남들이 1개 만들 때 우리는 10개.
```

### 2. 디지털 자아 구축 파이프라인
```
Parksy Capture (일상 캡처)
        ↓

        ↓
    ┌───┴───┐
    ↓       ↓

    ↓       ↓

    ↓       ↓
    └───┬───┘
        ↓

```

## 제품 정의

**dtslib.com Mobile-first AI Builder**는 비개발자가 스마트폰만으로 완전한 IT 인프라를 구축할 수 있게 해주는 **보일러플레이트 시스템**입니다.

**KOOSY**는 이 시스템의 첫 번째 브랜치 구현입니다.

```
dtslib.com Mobile-first AI Builder
│
├── 코어 인프라
│   ├── 6개 레포 네트워크 템플릿
│   ├── Staff Portal 프레임워크
│   ├── PWA/APK 빌드 파이프라인
│   └── 데이터 캡처 시스템 (Parksy)
│
├── 브랜치 구현
│   ├── KOOSY (콘텐츠 비즈니스) ← 첫 번째
│   ├── [다음 브랜치] ← config.json만 수정
│   └── [계열사 N] ← 무한 확장
│
└── AI 레이어
    ├── Claude Code 통합
    ├── RAG 지식 검색
    └── Fine-tuning (나만의 LLM)
```

## 왜 10배 빠른가?

### 기존 콘텐츠 제작 플로우
```
1. 경험/아이디어 발생
2. 메모장에 정리 (30분)
3. PPT/자료 제작 (2-3시간)
4. 촬영 환경 세팅 (1시간)
5. 촬영 (1시간)
6. 편집 (3-5시간)  ← 여기서 죽음
7. 자막 추가 (1시간)
8. 썸네일 제작 (30분)
9. 업로드
```
**총 소요: 1-3일**

### dtslib Builder 플로우
```
1. 경험/아이디어 발생
2. Parksy Capture로 캡처 (실시간)
3. 웹페이지 자동 생성 (즉시)
4. 태블릿 화면녹화 + 오버레이 (실시간)
5. 바로 업로드
```
**총 소요: 1-2시간**

### 왜 가능한가?

| 병목 | 기존 해결법 | dtslib 해결법 |
|------|------------|--------------|
| PPT 제작 | 수동 작업 | 웹페이지 자동 생성 |
| 편집 | 프리미어/다빈치 | **편집 안 함** (화면녹화) |
| 자막 | 수동/AI 자막 | 크롬 번역 자동 |
| 새 도구 필요 | 구매/학습 | Claude로 즉시 개발 |

**핵심:** 화면녹화 + 실시간 오버레이로 **후편집 자체가 불필요**

## 디지털 자아 = 장기 게임

### 데이터가 쌓이면 뭐가 되나?

**단기 (지금)**
- 콘텐츠 빠르게 찍어냄
- YouTube 수익화
- 포기하지 않고 버틸 수 있는 구조

**중기 (1-2년)**
- Parksy로 쌓인 마크다운 데이터
- RAG 시스템 구축
- "내가 예전에 뭐라고 했지?" → 검색 가능

**장기 (3년+)**
- 축적된 데이터로 Fine-tuning
- 나만의 LLM 생성
- **디지털 자아 복제**
- 멀티 에이전트로 분신술

### 왜 이게 중요한가?

```
일반 유튜버: 영상만 남음, 지식은 휘발
dtslib 유저: 모든 경험이 구조화된 데이터로 영구 축적
              → 검색 가능
              → 학습 가능
              → 복제 가능
              → 양도 가능 (Exit 시 가져감)
```

## 시스템 아키텍처

### 6개 레포지토리 네트워크

| 레포 | 용도 | 호스팅 |
|------|------|--------|
| **main-branch** | 랜딩 + Staff Portal | GitHub Pages |
| **posts** | HTML 블로그 | GitHub Pages |
| **logs** | Parksy 데이터 축적 | GitHub API |
| **media** | 이미지/음원 CDN | jsDelivr |
| **pwa** | 프로그레시브 웹앱 | Vercel |
| **apk-lab** | 안드로이드 APK | GitHub Actions |

### 왜 GitHub 중심인가?

```
전통적 아키텍처:
사용자 → 서버 → DB → 응답
         ↓
    월 수만원
    서버 장애 대응
    스케일링 고민

dtslib 아키텍처:
사용자 → GitHub Pages → 즉시 응답
         ↓
    비용 0원
    Microsoft가 관리
    무한 스케일
```

### 모바일 퍼스트 = PC 불필요

- **Termux + Claude Code** = 스마트폰에서 풀스택 개발
- **Parksy Capture** = 어디서든 데이터 캡처
- **화면녹화** = 태블릿에서 바로 콘텐츠 제작

**PC? 선택사항.**

## 비즈니스 모델

### Collaborator 모델

```
본사 (dtslib.com)
│
├── Collaborator로 멤버 등록
├── 인프라 제공 (6개 레포 세팅)
├── AI 개발 지원 (Claude Code)
├── 데이터 파이프라인 구축
│
└── Exit 시: 누적 수익의 30%로 독립
    ├── 모든 레포 Fork (완전한 소유권)
    ├── 축적된 데이터 가져감
    └── 독립 개발 가이드 제공
```

### 왜 턴키가 아닌가?

**실제 테스트 결과:**
```
친구에게 턴키로 전달 시도
→ GitHub 가입 → OK
→ Termux 설치 → OK
→ Claude Code 연동 → 막힘
→ 결론: 실패
```

GitHub + Termux + Claude Code 체인 중 **하나만 막혀도 끝**
비개발자에게 가르쳐서 혼자 하게 하는 건 **불가능**

**결론:** Collaborator로 같이 키우고, 성공하면 30%로 독립
